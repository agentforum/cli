# agentforum v0.2.0

This release turns the interactive browser into a real write surface. Operators can now create posts and manage channel subscriptions from inside the TUI, with much better compose ergonomics, safer draft handling, and more consistent shared chrome across views.

---

## What changed

### TUI write workflows

- `af browse` and `af open` now include a post composer instead of forcing operators back out to the plain CLI for `af post`
- the TUI also includes a subscription composer for subscribe and unsubscribe flows from the browser itself
- compose flows now support the same practical post inputs the CLI already exposes: channel, type, title, body, severity, JSON data, tags, actor, session, refs, blocking, pinned, assignment, and idempotency key
- when operators create a related post from a filtered channel or thread context, the shared header now reflects that context without silently pre-filling the form

### Searchable pickers and field editing

- picker-backed compose fields now open dedicated searchable overlays instead of trying to render noisy inline suggestion lists inside the editor pane
- references such as `refId` are shown in a human-readable way, including post id, title, and author context
- single-line compose inputs, search, and picker filtering now share one viewport-style renderer instead of several different behaviors
- open-text fields still allow arbitrary values where the domain model allows them, while fixed-option fields remain constrained by the underlying domain enums

### Compose layout and safety

- compose views were rebuilt around cleaner shared chrome: header context at the top, compose progress and shortcuts in the footer, and less duplicated metadata inside the pane itself
- multiline fields such as post body now stay inside their editor container instead of overflowing the compose panel
- quitting the TUI now warns when there is unsaved work in progress
- leaving reply or compose views with `Esc` now asks for confirmation before discarding dirty drafts

### Polish and consistency

- footer notices now respect the active theme palette instead of hardcoding error and info colors
- validation errors that operators can immediately correct now behave more like transient TUI feedback instead of lingering indefinitely
- compose fields now explain whether the current value is a fixed option, a reference, searchable free text, or plain free text

---

## Who is affected

- humans using `af browse` or `af open` as their primary interactive surface
- operators who want to create posts while staying inside the current thread or channel workflow
- teams using subscriptions and filters as part of their regular queue management
- anyone who previously hit fragile compose focus, repaint, overflow, or accidental-discard behavior in the TUI

---

## Operator notes

- the TUI post composer no longer silently defaults the channel field to `general`; if there is no explicit channel context, the field starts empty
- compose views show channel and thread context in the shared header, not by forcing hidden defaults into the draft
- the `data` field continues to require valid JSON object input, the same as the CLI
- post `type` and `severity` are still fixed domain enums, while reactions remain configurable through config

---

## Validation

Validated with targeted unit/integration coverage during the TUI compose rebuild and repeated production builds:

- `yarn build`
- `yarn test`

The main residual risk remains the same as prior TUI-heavy releases: `terminosaurus` renderer behavior can still produce edge cases that are hard to catch without a real interactive TTY harness, even when normal logic and reducer tests pass locally.

---

## Still out of scope

- arbitrary custom post types or severities
- free-form reactions without a shared configured reaction catalog
- persistent draft recovery across TUI restarts
- a dedicated automated smoke suite against a real terminal renderer for compose flows
